## TL;DR — Executive Summary (non-technical)

**You have a real product here.** The bones are good: Supabase backend, real auth, an AI edge function, a non-trivial plan generator, PDF export, plan sharing, ingredient substitution, progress tracking, exercise video library, notifications. The README claims 4 "market-leading features" and the architecture genuinely supports them.

**But it's not sellable as-is.** It's a Lovable-generated prototype with critical engineering debt baked in:

- 4 service files over 350 lines (one is **1,147 lines**), 2 hooks over 600 lines — clear monolith anti-patterns that block any new contributor from being productive.
- A complete data layer (progress tracking, notifications, sample data) lives in **`localStorage`**, not the database. The user has no way to know this.
- The `useNutritionPlanState` hook is 688 lines, has its own test file of 610 lines — it's a state machine pretending to be a hook.
- A 7-day meal plan + training plan is generated by **random pick of the first suitable recipe** (see `planGenerator.ts:134`).
- No CI workflow is present despite the README badge.
- No error boundaries. No telemetry. No rate limiting on AI calls. No cost guardrails.
- The "AI breakthrough" claim is a single edge function. There's no real differentiation yet.

**The good news:** the path from "working prototype" to "shippable SaaS" is **2–4 weeks of focused engineering**, not months. Most of the infrastructure is already in place — Supabase, auth, types, a clear domain model. The work is mostly **consolidation, hardening, and 3–5 high-impact product features**.

**The breakthrough opportunity:** the market is flooded with calorie counters (MyFitnessPal, Cronometer) and generic AI coaches. The genuinely underserved niche is **coach-mediated accountability** — a B2B tool for independent nutrition coaches to manage 20–200 clients with AI-assisted plan generation, automatic progress dashboards their clients can see, and white-label client portals. That's where the value is, and your codebase is already pointed in that direction (`ClientPage`, `clientInvitationService`, `SharePlanButton`, multi-client routing).

## 2. Detailed Technical Review

### 2.1 Strengths (keep these)

1. **Supabase-first architecture** — auth, edge functions, migrations all wired correctly. The migration history shows real iteration (10+ migrations, dated through 2026-02-04).
2. **Clear domain types** (`src/types/index.ts`) — `Client`, `MealPlan`, `TrainingPlan`, `CompletePlan` are well-shaped. The discriminator unions on `dietType`, `primaryGoal`, `sessionType` are correct.
3. **Graceful AI degradation** — `planGenerator.ts:530` falls back to local generation on edge function failure. This is rare and excellent.
4. **Red flag detection** (`planGenerator.ts:710`) — refuses to generate plans for dangerous medical conditions. Important for a health app.
5. **Test coverage on the most complex piece** — `useNutritionPlanState` has dedicated tests (1077 lines of tests for 688 lines of hook = healthy ratio).
6. **PDF export** — already integrated via `jspdf` + `html2canvas`.
7. **Plan sharing** via shareable links — `sharePlanService` + `PlanViewerPage` route exist.

### 2.2 Critical issues (must fix)

#### 🔴 C1. `localStorage` is being used as a database
- `ProgressTracker.tsx:34` — `localStorage.getItem(\`progress_${clientId}\`)`
- `NotificationCenter.tsx` (312 lines) — settings + history in localStorage
- `MealSwapper.tsx:169` — likely also localStorage
- **Impact:** User data is browser-bound. Switching devices = total data loss. No multi-device sync. This single issue blocks any "selling product" claim.
- **Fix:** migrate to Supabase tables. ~3 days of work.

#### 🔴 C2. Service files are 350–1147 lines
- `services/recipeService.ts` — **1,147 lines** (largest)
- `hooks/useNutritionPlanState.ts` — 688 lines
- `services/planService.ts` — 510 lines
- `hooks/useNutritionPlanLifecycle.test.ts` — 467 lines
- `services/supabasePlanService.ts` — 362 lines
- **Impact:** Unmaintainable. A new dev can't review in one sitting. No way to test pieces in isolation. Merge conflicts guaranteed.
- **Fix:** split by domain. See §4 refactor plan.

#### 🔴 C3. No error boundaries anywhere
- Search confirms: zero `ErrorBoundary` in the codebase.
- **Impact:** A single rendering error in `ProgressTracker` or `WeeklyMealPlanDisplay` takes the whole app down to a white screen.
- **Fix:** add one root boundary + one per page boundary. ~2 hours.

#### 🔴 C4. Recipe selection is random-first
- `planGenerator.ts:134` — `let bestRecipe = mealRecipes[0];` then iterates to find a "better" one. But `Math.random()` in `recipeService.ts:78` selects proteins randomly. Combined, the AI's "best" plan is partly random.
- **Impact:** Plans are non-deterministic. Same client, same input → different plan each generation. Coaches can't reproduce or iterate. Users will report "the AI gave me a worse plan last time."
- **Fix:** deterministic scoring (already partially there), seed the RNG with `client.id` + date, or make it a proper convex optimization.

#### 🔴 C5. No CI
- README references `actions/workflows/ci.yml` (line 1 badge) — **the workflow does not exist** in `.github/workflows/`.
- **Impact:** Every push could be broken. No typecheck, no lint, no test gate.
- **Fix:** add a minimal CI in 30 minutes (see §5).

#### 🔴 C6. Types sent to edge function are not validated
- `planGenerator.ts:504` sends the entire `Client` object to the edge function. The Client type has 30+ fields, many optional.
- **Impact:** Bad data → bad AI output. No schema check. Edge function probably re-validates (or doesn't).
- **Fix:** Zod schema at the edge. ~3 hours.

### 2.3 High-priority issues

#### 🟠 H1. No tests on the core domain logic
- `ingredientSubstitution.ts`, `planGenerator.ts`, `calculations.ts` (the actual algorithms) are untested. The 2 test files test the state machine, not the math.
- **Impact:** A 5-line change in macro distribution can silently break the entire app's core value prop.
- **Fix:** snapshot tests for plan generation, unit tests for substitution scoring. ~2 days.

#### 🟠 H2. No pagination on clients/plans
- `useSupabaseClients.ts` (156 lines) likely loads all clients. With 50+ clients a coach will hit issues.
- **Fix:** add pagination or virtualization. ~1 day.

#### 🟠 H3. `equipment` and `equipmentAvailable` are duplicate fields
- `Client` type has both `equipment: string[]` AND `equipmentAvailable?: string[]`. Dead code, guaranteed to drift.
- **Fix:** pick one, migrate data, delete the other.

#### 🟠 H4. No telemetry / observability
- `console.error` is the only error tracking. A paying user hitting a 500 = invisible.
- **Fix:** Sentry (free tier) or PostHog. ~1 day.

#### 🟠 H5. Bundle size likely bloated
- Dependencies include: `html2canvas` (300KB), `jspdf` (400KB), `recharts` (200KB), `date-fns`, `react-day-picker`, `embla-carousel-react`, `vaul`, `cmdk`, `input-otp`, **and** `next-themes` (no Next.js!). Some are likely unused.
- **Fix:** audit with `vite-bundle-visualizer`. ~half a day.

### 2.4 Product / UX gaps

#### 🟡 P1. No onboarding
- A new coach lands on `ClientPage` (per `App.tsx:36`). They have zero clients. Empty state is likely a blank screen.
- **Fix:** 3-step onboarding flow (welcome → create first client → generate first plan).

#### 🟡 P2. Client portal is `PlanViewerPage` only
- The share link shows a plan. The client can't log adherence, can't message the coach, can't see their own progress charts.
- **Fix:** client-side auth (already in routes), client dashboard. ~1 week.

#### 🟡 P3. No billing / plan tier
- No Stripe. No tier logic. Can't sell premium.
- **Fix:** Stripe Checkout + customer portal. ~2 days.

#### 🟡 P4. No mobile-first view for clients
- App is desktop-tuned. Most clients will open on phone.
- **Fix:** dedicated mobile layout for `PlanViewerPage` and client dashboard.

#### 🟡 P5. Internationalization
- All hardcoded English. French/European market is your natural audience (README hints at this with "Poulet Grillé" in the strategy doc).
- **Fix:** i18next or react-intl. ~3 days for first locale.

## 4. Target Architecture

**Pattern: Modular Monolith** (not microservices, not pure Clean Architecture).

Why: you have one app, one team, one deploy. Microservices are organizational debt you don't need. Clean Architecture is overkill for a 9k-line codebase and adds indirection. Modular Monolith gives you the boundaries without the cost.

```
src/
├── app/                    # Bootstrap: App.tsx, providers, router
├── pages/                  # Route components, thin
├── features/               # Vertical slices — each owns its UI, state, types
│   ├── clients/
│   │   ├── components/
│   │   ├── hooks/
│   │   ├── services/       # clientService.ts (≤200 lines)
│   │   ├── types.ts
│   │   └── index.ts        # public API
│   ├── plans/
│   │   ├── components/
│   │   ├── hooks/          # split useNutritionPlanState into 4 hooks
│   │   ├── services/       # split recipeService.ts
│   │   ├── domain/         # planGenerator, substitution — PURE functions
│   │   └── index.ts
│   ├── progress/
│   ├── notifications/
│   ├── ingredients/
│   ├── training/
│   └── sharing/
├── shared/                 # Cross-feature primitives
│   ├── ui/                 # shadcn components (already there)
│   ├── hooks/              # useAuth, useToast, use-mobile
│   ├── lib/                # utils.ts
│   └── types/              # global types only
└── integrations/
    └── supabase/           # client, generated types
```

**Rules:**
1. `features/*` cannot import from another `features/*` directly — go through `index.ts`.
2. `domain/` files have **zero React imports** — testable in node, edge-runnable.
3. Service files cap at 200 lines. If longer, split.
4. Hook files cap at 150 lines. If longer, split.

**Migration cost:** ~1 week for one dev, while keeping the app running.

## 6. The "Breakthrough" — what makes this app win

Generic AI coaching apps (betterme, fitnessai, etc.) lose to **MyFitnessPal + ChatGPT** because they don't have a moat. Your moat is **coach-mediated accountability**. Lean into it.

### 6.1 The 3 things no competitor does well

1. **Coach sees the client's real adherence** — clients log daily, coach sees streaks/breaks, can intervene before a relapse.
2. **AI does the busywork, coach does the human work** — coach edits 5% of the plan, AI regenerates the rest. Most apps are 100% AI (no human) or 100% manual (no AI). The hybrid wins.
3. **White-label client portal** — coach can put their own branding on the client view. Coaches who build a personal brand will pay 3x for this.

### 6.2 The 2 things to NOT build

- ❌ A general-purpose calorie tracker. Cronometer exists. MyFitnessPal exists. You lose.
- ❌ A social feed. Strava exists. Building community is a 10-year play.
- ❌ Mobile native app. PWA is enough for v1. Native = $30k+ in dev costs you don't need yet.

### 6.3 The 5-year positioning

> "The CRM for nutrition coaches."
>
> If a coach charges $200/month and has 50 clients = $10k MRR. Your tool at $49/month = their cost is 0.5% of revenue. They'll pay. They'll never leave because the client data is their business asset.
